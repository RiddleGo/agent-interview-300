# 第 48 题：提示链（Prompt Chaining）与单次复杂提示的对比

## 题目

什么是提示链（Prompt Chaining）？与单次复杂提示的对比？

---

## 一、什么是提示链？

**提示链（Prompt Chaining）** 指把**一个复杂任务拆成多步**，每步用**一次模型调用**完成一个子任务，上一步的**输出**作为下一步的**输入**（或部分输入），形成“调用1 → 输出1 → 调用2(含输出1) → 输出2 → …”的链。每步的 prompt 可不同、可针对该步专门设计，整体由应用逻辑串联。

---

## 二、与单次复杂提示的对比

| 维度 | 单次复杂提示 | 提示链 |
|------|--------------|--------|
| **调用次数** | 1 次 | 多次，步数=调用数 |
| **单次 context** | 需容纳全任务与中间信息 | 每步只含该步所需 |
| **可控与可维护** | 一大段 prompt 难维护 | 每步独立 prompt，易迭代与 A/B |
| **错误定位** | 难确定错在哪一环节 | 可按步检查与重试 |
| **延迟与成本** | 单次长 context、一次计费 | 多次调用、总 token 可能更多 |
| **中间结果** | 仅在模型“内心” | 每步有明确输出，可存可审 |

**单次复杂提示**：适合任务简单或强依赖“一口气推理”的场景；一大段 prompt 易超出窗口、难调试。  
**提示链**：适合可自然拆成子任务、每步有清晰输入输出的流程（如：查询改写 → 检索 → 摘要 → 回答）；可维护性与可观测性更好，但延迟与调用次数增加。

---

## 再深入一层：链式 = 显式状态机

- **状态与 schema**：每步输出建议用 **JSON** 固定字段（如 `rewritten_query`, `evidence_ids`），下一步只读字段子集，避免「自然语言糊成一团」难以解析与单测。  
- **重试与补偿**：某步失败（检索空、解析错）时，链上应有**降级分支**（换检索策略、缩小问题、转人工），而不是反复同一 prompt。  
- **与 Agent 循环**：提示链多是**DAG 或有限步流水线**；ReAct 式 Agent 是**带工具的循环**；面试可对比「可控编排 vs 自主扩展」。

## 常见追问

- **token 总成本一定更高吗？** 不一定：每步 context 更短、可缓存系统前缀；但**首字节延迟**通常随步数累加，要做并行化（如检索与意图分类并行）。  
- **如何调试？** 为每步打 **trace id**、落库输入输出与耗时，Bad Case 可精确定位到步。  
- **何时退回单次 prompt？** 强依赖全局连贯创作、或子步之间无法定义清晰接口时。

---

## 三、小结与面试要点

**小结**：提示链 = 多步调用、上步输出为下步输入，每步独立 prompt；与单次复杂提示比，更易维护与排错、可观测性好，但调用次数与总延迟增加；适合可拆成清晰子任务的工作流。进阶上要谈步间 schema、降级分支与和 Agent 的差异。

**面试要点**：

- 提示链：多步调用，上步输出→下步输入，每步独立 prompt。  
- 对比单次：单次一大段难维护难排错；链式易维护、可观测，但多次调用。  
- 适用：可拆成子任务的工作流（如改写→检索→摘要→回答）。  
- 进阶：JSON 状态、降级/重试、trace、与 Agent 循环对比、延迟与并行。

---

## 记忆要点

1. 提示链 = 多步调用，上步输出→下步输入。  
2. 优点：易维护、可观测、易排错；缺点：多次调用、延迟增加。  
3. 适合可拆成子任务的工作流。
