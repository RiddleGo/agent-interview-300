# 第 52 题：结构化提示（Structured Prompting）与 JSON/XML 模式的作用

## 题目

什么是结构化提示（Structured Prompting）？JSON/XML 模式的作用是什么？

---

## 一、什么是结构化提示？

**结构化提示** 指用**固定格式与标签**组织 prompt 和/或要求输出——如用 XML 标签区分“指令”“上下文”“示例”“当前输入”，或用 JSON 规定“必须返回某 schema 的 JSON”。目的是便于**解析、校验与下游对接**，并减少模型输出格式漂移。

---

## 二、JSON/XML 模式的作用

- **输入侧**：用 XML/JSON 把 prompt 分成清晰区块（如 `<instruction>...</instruction>`、`<context>...</context>`、`<examples>...</examples>`），模型更容易识别“哪里是指令、哪里是背景、哪里是示例”，提升遵循度。  
- **输出侧**：要求模型**只输出 JSON**（并给出 schema 或示例），便于程序解析、校验必填字段与类型、写入 DB 或调 API；减少自由文本导致的解析失败。  
- **约束**：schema 可标出必填、枚举、长度限制，配合后处理校验，降低幻觉与越界输出。  
- **多轮与工具**：Agent 中常用 JSON 表示“工具名+参数”，与 Function Calling 对齐；XML 常用于多轮对话的段落划分。

---

## 再深入一层：结构化之后还有「修复层」

- **模型仍可能违 schema**：要在网关做 **JSON Schema 校验**、缺字段自动拒收或触发「自我纠错」重试；不要假设一次生成永远合法。  
- **XML 注入**：若用户输入含 `</instruction>` 等闭合标签，可能**提前结束**指令区；需对用户内容转义或包进 CDATA/不可解析占位。  
- **与 constrained decoding**：强 schema 场景可用 **grammar-guided decoding**（如 outlines、jsonformer）把「合法 JSON」硬约束在解码器侧，减少后处理。

## 常见追问

- **JSON vs XML 选型？** JSON 更贴 API/JS 栈、体积小；XML 更适合**人类可读的大段分层**与富文本包裹；也可混用（外层 XML 分区、内层 JSON 载荷）。  
- **如何测格式稳定性？** 构造「对抗性用户输入 + 长上下文 + 多语言」压测，统计 **schema 通过率** 与修复轮数。  
- **和「工具调用」关系？** FC 的 `arguments` 本质是结构化输出；提示里用同一 schema 描述，减少双份维护。

---

## 三、小结与面试要点

**小结**：结构化提示 = 用固定格式与标签组织输入/输出；JSON/XML 用于区分区块、规定输出 schema、便于解析与校验、约束输出边界；Agent 与生产场景常用 JSON 输出、XML 组织长 prompt。进阶上要谈校验/修复、XML 转义与约束解码。

**面试要点**：

- 结构化提示：固定格式与标签，便于解析与约束。  
- JSON/XML：输入分区、输出 schema、解析与校验、约束边界。  
- 作用：减少格式漂移、便于下游对接与校验。  
- 进阶：schema 校验与重试、XML 注入、grammar-guided decoding、与 FC 对齐。

---
