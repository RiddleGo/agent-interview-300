# 第 59 题：提示版本管理及在生产环境中的实践

## 题目

什么是提示版本管理？在生产环境中如何实践？

---

## 一、什么是提示版本管理？

**提示版本管理** 指对 prompt 的**内容、模板与配置**做**版本化**（如 Git、数据库版本号、配置中心），支持**追溯、回滚、对比与审计**，并与**发布、灰度、A/B 测试**结合，使提示的变更可控、可复现、可评估。

---

## 二、生产实践要点

- **存储与版本**：将 prompt 模板与变量放在**代码库**（如 YAML/JSON 文件）或**配置中心**，用版本号/commit 标识；变更走 MR/审批，避免直接改生产配置。  
- **环境分离**：**开发/预发/生产**使用不同配置或同一配置的不同版本，避免实验提示直接影响线上。  
- **灰度与 A/B**：新提示先**灰度**到小流量或**A/B 测试**（按用户或请求分流），用指标（准确率、满意度、安全事件）决定是否全量；保留旧版本以便回滚。  
- **可观测**：记录**请求对应的 prompt 版本**（或模板 id）、模型与参数，便于按版本分析 Bad Case、延迟与成本。  
- **回滚**：一旦新版本异常，可**一键回滚**到上一版本或指定版本，无需重新发代码。  
- **文档与评审**：重要提示变更附**变更说明与预期效果**，做同行评审；与产品/安全对齐后再上线。

---

## 再深入一层：把 prompt 当成「配置发版」而不是聊天记录

- **语义化版本**：对对外承诺行为变更的提示用 **MAJOR** 升级（如拒答策略收紧）；纯措辞优化用 **MINOR**；变量默认值修正用 **PATCH**；便于业务方理解兼容性。  
- **与模型版本矩阵**：同一 `prompt_v3` 换基座模型可能行为漂移；要维护 **(model_id, temperature, prompt_version)** 三元组兼容性表，上线 checklist 一起测。  
- **紧急回滚**：除配置回滚外，准备 **「熔断到上一稳定 triple」** 的自动化（监控到安全/格式 SLO 跌破阈值即触发）。

## 常见追问

- **存 Git 还是数据库？** Git 适合**文本 diff + MR**；数据库适合**运营热更**与多租户；常见是 Git 为 source of truth，发布管道同步到 **KV/配置中心**。  
- **如何做 diff 评审？** 高亮「系统指令变更」与「示例变更」分开审；安全相关走 **双人复核**。  
- **和 feature flag 关系？** `prompt_version` 可作为 flag 负载的一部分，实现按租户/地区灰度。

---

## 三、小结与面试要点

**小结**：提示版本管理 = 对 prompt 做版本化与变更控制；生产中通过存储与版本、环境分离、灰度与 A/B、可观测、回滚与文档评审实践，保证变更可控、可追溯、可回滚。进阶上要联系语义化版本、与模型/解码参数的兼容矩阵、熔断与 Git+DB 分工。

**面试要点**：

- 定义：prompt 内容/模板的版本化，支持追溯、回滚、对比。  
- 实践：存储与版本、环境分离、灰度 A/B、可观测、回滚、文档与评审。  
- 价值：变更可控、可复现、可评估与回滚。  
- 进阶：semver、模型矩阵、自动熔断、Git vs 配置中心、与 feature flag。

---
