# 第 167 题：身份认证与 SSO 集成

## 题目

Agent 应用如何与企业身份认证、SSO 集成？

---

## 一、集成目标

- **统一登录**：用户**一次登录**企业 SSO（如 SAML、OAuth2/OIDC），即可访问 Agent 应用，无需再输账号密码。  
- **身份传递**：请求中带**身份信息**（user_id、tenant_id、角色）；Agent 与下游系统按身份做**鉴权与数据隔离**。  
- **合规**：满足**审计**（谁在何时访问）、**会话超时**与**强制登出**等策略。

---

## 二、常见方式

- **OAuth2 / OIDC**：Agent 前端或网关作为 **RP**，向企业 **IdP**（如 Azure AD、Okta、Keycloak）请求 **Authorization Code** 或 **Token**；用 **Access Token** 或 **JWT** 调用 Agent API；后端**校验签名与 claims**，解析 user_id、tenant_id、角色。  
- **SAML**：企业 IdP 发起 **SAML 断言**；Agent 侧 **SP** 校验断言并建会话；适合传统企业 SSO。  
- **API Key + 用户映射**：企业**预先同步**用户与 API Key 或内部标识；请求带 Key，后端查表得到 user_id/tenant_id；适合服务间或简化场景。  
- **网关统一认证**：在 **API 网关** 层做 **JWT 校验、OAuth 代理**；通过后**注入身份 Header** 传给 Agent 服务；Agent 只信任网关注入的身份。

---

## 三、实现要点

- **Token 校验**：**签名、过期、issuer/audience** 校验；**刷新** Access Token 时用 Refresh Token 或静默重定向。  
- **会话**：**服务端会话**（如 Redis）存 user_id、tenant_id、权限；**过期与登出**时清理；或**无状态**仅校验 JWT 每请求。  
- **多租户**：JWT 或 Session 中带 **tenant_id**；下游 API 与数据访问**强制带租户**过滤。  
- **审计**：**登录、登出、Token 使用**记审计日志；与企业管理平台对接。

---

## 四、小结与面试要点

**小结**：与企业 SSO 集成常用 OAuth2/OIDC 或 SAML；实现需 Token 校验、会话与过期、多租户传递与审计；网关统一认证可集中处理。

**面试要点**：能说清 OAuth2/OIDC 与 SAML 的适用场景；Token 校验与身份传递要提到；多租户与审计要举一二。

---

## 记忆要点

1. 方式：OAuth2/OIDC、SAML、API Key 映射、网关统一认证。  
2. 实现：Token 校验、会话与过期、多租户、审计。  
3. 目标：统一登录、身份传递、合规。
