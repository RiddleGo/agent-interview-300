# 第 177 题：跨系统数据格式

## 题目

Agent 与多系统集成时，如何约定与处理跨系统数据格式？

---

## 一、常见格式

- **JSON**：**REST API** 与**消息**常用；**Schema** 可用 **JSON Schema** 描述；便于嵌套与扩展；注意**数字精度、日期**的跨语言一致。  
- **Protocol Buffers**：**二进制**、**强类型**、**向后兼容**；适合**高性能、多语言**的 gRPC 与内部 RPC；需维护 **.proto** 与生成代码。  
- **XML**：传统 **SOAP、企业接口** 仍常见；**命名空间与 Schema** 严格；与 LLM 文本交互时需**序列化/反序列化**。  
- **表格/CSV**：**导出、报表、批量**；**分隔符与编码**统一（如 UTF-8）；大文件需**流式**处理。  
- **事件**：**CloudEvents** 等**统一事件格式**（id、source、type、data）；便于**跨系统、跨云**事件总线。

---

## 二、约定与兼容

- **Schema 先行**：**API 与消息**先定 **Schema**（字段、类型、必选/可选）；**版本**（如 v1、v2）在 path 或字段中体现；**文档**与**示例**同步。  
- **兼容策略**：**向后兼容**（新字段可选、旧客户端忽略）；**废弃**字段先标记 deprecated 再下版移除；**不兼容变更**用**新版本 API**。  
- **编码与时间**：**统一 UTF-8**；**时间**用 **ISO 8601** 或 **Unix 毫秒**；**时区**明确（如 UTC）。  
- **空与缺失**：**null、空串、缺失**语义一致；**可选字段**用 **optional** 或约定“不传即默认”。

---

## 三、与 Agent 结合

- **Tool 输入输出**：**Tool** 的**参数与返回**用 **JSON Schema** 或 **OpenAPI** 描述；LLM 生成 **JSON** 后**校验**再调 Tool；**错误**时返回**结构化错误码与 message**。  
- **多源统一**：**适配层**将**各系统**返回转为**统一内部格式**（如统一 Message、Document）；Agent 与 RAG 只消费统一格式，减少分支。  
- **大对象**：**大文本、二进制**用 **URL、OSS 引用** 或**分页**；避免单次 payload 过大。

---

## 四、小结与面试要点

**小结**：跨系统常用 JSON、Protobuf、XML、事件格式；需 Schema 先行、版本与兼容策略、编码与时间统一；Agent 侧 Tool 用 Schema 校验、多源经适配层统一格式。

**面试要点**：能举 2～3 种格式及适用场景；向后兼容与版本策略要提到；Tool 的 Schema 校验要强调。

---

## 记忆要点

1. 格式：JSON、Protobuf、XML、事件（CloudEvents）。  
2. 约定：Schema 先行、版本、兼容、编码与时间。  
3. Agent：Tool Schema 校验、适配层统一、大对象引用。
