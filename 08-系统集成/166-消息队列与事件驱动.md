# 第 166 题：消息队列与事件驱动

## 题目

Agent 应用如何与消息队列、事件驱动架构集成？

---

## 一、使用场景

- **异步任务**：用户请求**入队**，由 Worker 消费并调用 Agent；避免 HTTP 超时，支持**重试与削峰**。  
- **事件驱动**：业务系统**发布事件**（如工单创建、审批完成、数据变更），Agent 或下游**订阅**后触发处理（如自动回复、知识库更新、通知）；解耦与扩展性好。  
- **多 Agent 协作**：Agent 间通过**消息**传递任务与结果；队列作为**缓冲与路由**，支持负载均衡与背压。  
- **日志与审计**：将**操作与调用**发往 **Kafka** 等，供离线分析、审计与实时告警。

---

## 二、集成要点

- **协议与格式**：消息 **payload** 统一格式（如 JSON）；含 **event_type、trace_id、payload、timestamp**；版本与兼容策略明确。  
- **消费语义**：**至少一次**（at-least-once）需**幂等**处理；**精确一次**（exactly-once）需事务或去重；**死信队列**处理多次失败后的告警与人工介入。  
- **与 Agent 结合**：Worker 从队列取任务 → 调 Agent API → 将结果写回 DB 或发新事件；**长任务**可先返回 task_id，结果通过 **Webhook 或轮询** 回传。  
- **运维**：**延迟、积压、错误率**监控；队列与 Worker 的**扩缩容**与**优雅关闭**（排空再停）。

---

## 三、常见组件

- **Kafka**：高吞吐、持久化、多订阅者；适合事件流与日志。  
- **RabbitMQ**：灵活路由、多种模型；适合任务队列与 RPC。  
- **Redis Streams / 云队列**：轻量、延迟低；适合简单任务队列与限流。  
- **云厂商**：AWS SQS/SNS、阿里云 MNS 等；托管、与云生态集成。

---

## 四、小结与面试要点

**小结**：消息队列用于异步任务、事件驱动、多 Agent 协作与审计；集成需统一消息格式、消费语义与幂等、死信与监控；常见 Kafka、RabbitMQ、Redis；Agent 常作为消费者或结果回写方。

**面试要点**：能举 2～3 个场景（异步、事件驱动、多 Agent）；消费语义与幂等要提到；能举 1～2 个队列组件。

---

## 记忆要点

1. 场景：异步任务、事件驱动、多 Agent、审计。  
2. 要点：消息格式、消费语义与幂等、死信、监控。  
3. 组件：Kafka、RabbitMQ、Redis/云队列。
