# 第 150 题：API 设计规范

## 题目

LLM/Agent 对外 API 设计有哪些规范与最佳实践？

---

## 一、接口形态

- **OpenAI 兼容**：采用与 **OpenAI Chat Completions** 一致的路径与请求/响应格式（如 `/v1/chat/completions`、`messages`、`stream`）；便于客户端与生态复用、便于切换不同后端。  
- **RESTful**：资源清晰（如 `/v1/chat`、`/v1/embeddings`、`/v1/agents/{id}/invoke`）；用 HTTP 方法、状态码与 body 表达语义。  
- **流式**：支持 **stream=true** 时用 **SSE** 或 **chunked** 返回；约定 event 格式（如 `data: {"choices":[...]}`）与结束标记。  
- **同步与异步**：长任务可提供 **async** 接口（提交返回 task_id，再轮询或 webhook 取结果），避免 HTTP 超时。

---

## 二、请求与响应

- **认证**：**API Key**（Header）或 **OAuth**；多租户时带 **tenant_id** 或由 Key 推导。  
- **幂等**：写操作支持 **idempotency_key**，重复请求返回同一结果，便于重试。  
- **限流**：**Rate Limit** 头（如 `X-RateLimit-Remaining`）；429 时带 Retry-After。  
- **响应**：统一结构（如 `{ data, error, request_id }`）；错误码与 message 清晰；**request_id** 便于追踪与排查。  
- **版本**：路径或 Header 带 **API 版本**（如 `/v1/`），便于兼容与演进。

---

## 三、Agent 与 RAG 特有

- **会话**：若支持多轮，请求带 **session_id** 或 **conversation_id**；可选 **message_id** 做引用与去重。  
- **工具/来源**：若返回**工具调用**或**引用**，响应中明确 **tool_calls**、**citations**（chunk_id、doc_id、url）等结构。  
- **流式事件**：除 content delta 外，可约定 **event 类型**（如 `message_start`、`tool_use`、`citation`、`done`），便于前端分类型展示。

---

## 四、小结与面试要点

**小结**：API 设计可采用 OpenAI 兼容或 RESTful；支持流式与可选异步；请求需认证、幂等与限流，响应需统一结构与 request_id；Agent/RAG 需会话与引用等扩展字段。

**面试要点**：能说清 OpenAI 兼容的价值；能举 2～3 条规范（认证、限流、request_id、版本）；Agent 的 session 与引用要提到。

---

## 记忆要点

1. 形态：OpenAI 兼容或 RESTful；流式 SSE；可选 async。  
2. 规范：认证、幂等、限流、统一响应、request_id、版本。  
3. Agent/RAG：session_id、tool_calls、citations、流式事件类型。
