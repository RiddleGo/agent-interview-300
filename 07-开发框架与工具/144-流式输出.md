# 第 144 题：流式输出实现

## 题目

LLM 与 Agent 应用如何实现流式输出（Streaming）？注意点有哪些？

---

## 一、流式的目的

**流式** = 模型**边生成边返回** token，而非等整句/整段生成完再一次性返回。好处：**首字延迟低**、用户感知更流畅、长回答时可提前看到部分内容；适合对话与长文生成。

---

## 二、实现层次

- **模型/推理层**：推理引擎（vLLM、TGI、OpenAI API）支持 **stream=True** 或 **Server-Sent Events (SSE)**，按 token 或 chunk 推送；应用层消费该流并转发或加工。  
- **协议**：常用 **HTTP SSE**（`Content-Type: text/event-stream`）或 **WebSocket**；每条 event 带一段文本或 JSON（如 `{"choices":[{"delta":{"content":"..."}}]}`）。  
- **应用层**：从推理服务**读流**，可做**后处理**（如敏感词过滤、引用注入）、**重试与超时**；再以 SSE/WebSocket 推给前端；若链中有多步（如 RAG 后再生成），可设计“先返回检索完成事件、再流式返回生成”等。

---

## 三、Agent 与多步

- **单轮生成**：直接透传模型流即可。  
- **多步 Agent**：每步（thought、action、observation、answer）可**分事件**推送，如 `event: thought`、`event: action`、`event: chunk`；前端按类型展示（思考过程、工具调用、最终回答流）。  
- **引用**：流式过程中可在**合适边界**插入“引用块”事件（如某段对应 chunk id），前端实时高亮来源。

---

## 四、注意点

- **背压**：下游消费慢时，避免缓冲区无限增长；可设**超时**或**丢弃**策略。  
- **错误**：流中若模型或网络出错，需**半途结束**并带错误信息；客户端需能区分“正常结束”与“异常结束”。  
- **幂等与重试**：流式请求一般**不重试**整流；可对**非流式** fallback 或仅重试连接阶段。

---

## 五、小结与面试要点

**小结**：流式 = 边生成边返回，降低首字延迟、提升体验；实现依赖推理层 stream、SSE/WebSocket 协议与应用层转发与加工；Agent 可多事件类型流式；需注意背压、错误与重试策略。

**面试要点**：能说清“模型 stream → 应用转发 → 前端 SSE/WS”；Agent 多步流式可举例（thought/action/chunk）；背压与错误处理要提到。

---

## 记忆要点

1. 目的：首字延迟低、体验好；实现：推理 stream + SSE/WS + 应用转发。  
2. Agent：多事件类型（thought、action、chunk）分步流式。  
3. 注意：背压、错误半途结束、流式一般不整流重试。
