# 第 115 题：RAG 重排序（Reranker）

## 题目

RAG 中重排序（Reranker）的作用是什么？常见实现方式有哪些？

---

## 一、为什么需要重排序

**检索阶段**通常先用**向量或 BM25** 召回较多个候选（如 top-100），追求**高召回**；但向量/BM25 的排序未必与“与 query 的**真实相关性**”完全一致，且可能混入噪声。**重排序**用更精细的模型（或规则）对**少量候选**再做一次打分与排序，提升**精确率**与最终注入 context 的质量，减少无关 chunk 占用 context、降低幻觉。

---

## 二、常见实现

- **交叉编码器（Cross-Encoder）**：将 **query 与每个候选 chunk 拼成一对**输入 BERT 类模型，输出一个相关性分数；对 query-chunk 交互建模充分，效果通常优于双塔向量的一次性打分，但需对 N 个候选跑 N 次前向，**延迟与成本较高**，一般只对 top-k（如 20～50）做。  
- **列表式 Reranker**：一次输入 query + 多个候选，输出每个候选的分数或排序；减少前向次数，适合批量精排。  
- **LLM 打分**：用 LLM 对“query + chunk”打分或排序；灵活、可结合指令，但延迟与成本高，多用于离线评估或小规模精排。  
- **规则与启发式**：关键词覆盖、BM25 分、元数据（如来源权威性、时间）加权；实现简单，可作为轻量补充或初筛。

---

## 三、使用方式

典型流程：**检索召回 top-N（如 100）** → **Reranker 精排 top-K（如 5～10）** → **将 top-K 拼入 prompt**。  
Reranker 与检索可**同模型**（如都用 BGE），但更常见是**专用 Reranker 模型**（如 BGE reranker、Cohere Rerank）做精排，效果更好。

---

## 四、小结与面试要点

**小结**：重排序 = 对检索得到的候选再做精细排序，提升注入 context 的相关性；常用交叉编码器或列表式 Reranker，也可用 LLM 或规则；流程上先召回较多、再精排较少。

**面试要点**：能说清“先召回后精排”的目的；交叉编码器效果好但算力成本高，一般只对 top-k 做；能举 1～2 个 Reranker 模型或产品。

---

## 记忆要点

1. 目的：提升注入 context 的相关性，减少噪声与幻觉。  
2. 方式：交叉编码器（query+chunk 对）、列表式、LLM 打分、规则。  
3. 流程：检索 top-N → Reranker 精排 top-K → 拼入 prompt。
