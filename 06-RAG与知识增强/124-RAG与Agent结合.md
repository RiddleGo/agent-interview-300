# 第 124 题：RAG 与 Agent 结合

## 题目

RAG 如何与 Agent 结合？检索增强在 Agent 中扮演什么角色？

---

## 一、RAG 在 Agent 中的角色

Agent 需要**感知与决策**：除了工具调用、多轮对话，还常需要**外部知识**来回答问题、校验事实、补充上下文。**RAG** 可作为 Agent 的**一种能力或工具**：在需要“查知识库”时触发检索，将检索结果作为上下文或工具返回，供 Agent 推理与生成下一步动作或最终回答。

---

## 二、结合方式

- **RAG 作为工具**：Agent 拥有“检索知识库”类工具；在规划中决定“先检索再回答”时调用该工具，传入 query（可能由 Agent 改写），工具内部执行检索+可选重排，将 top-k chunk 或摘要返回给 Agent，Agent 再基于结果生成回复或继续规划。  
- **RAG 在每轮注入**：在 Agent 的**每轮 prompt 组装**时，用当前用户消息（或 Agent 提炼的 intent）做一次 RAG 检索，将结果作为“知识上下文”固定注入；Agent 无需显式调用工具即可看到相关文档，适合“以知识为主”的对话。  
- **按需 RAG**：Agent 先判断**是否需要**查知识库（如用分类或 LLM 判断）；仅当需要时再调用 RAG 工具或触发检索，节省成本与延迟。  
- **多步 RAG**：Agent 可能**多轮检索**（如先检索概览再根据结果追问细节）；或先检索再调用其它工具（如计算、API），最后综合 RAG 与工具结果生成答案。

---

## 三、设计要点

- **引用与溯源**：Agent 基于 RAG 结果生成时，应**引用来源**（chunk/doc），便于用户与审计确认依据。  
- **检索 query 质量**：由 Agent 生成的子 query 可能更精准（如多轮中提炼出的关键问题）；也可用 Agent 做查询改写与扩展后再检索。  
- **与其它工具协同**：RAG 与代码执行、API 调用等工具并列；Agent 规划中决定“先 RAG 再计算”或“先 API 再 RAG 验证”等顺序与组合。

---

## 四、小结与面试要点

**小结**：RAG 在 Agent 中可作工具或每轮注入的上下文；按需触发与多步 RAG 常见；设计上注意引用、query 质量及与其它工具的协同。

**面试要点**：能说清“RAG 作为工具”与“每轮注入”两种形态；按需与多步 RAG 要能举例；引用与 query 质量要提到。

---

## 记忆要点

1. 角色：RAG 作为知识能力/工具或每轮上下文。  
2. 方式：RAG 工具、每轮注入、按需触发、多步检索。  
3. 要点：引用溯源、检索 query 质量、与其它工具协同。
