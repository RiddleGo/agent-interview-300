# 第 121 题：RAG 上下文窗口与压缩

## 题目

RAG 中如何应对上下文窗口限制？有哪些上下文压缩策略？

---

## 一、问题背景

模型有**最大 context 长度**（如 8K、32K、128K token）；RAG 需同时放入：系统指令、检索到的多个 chunk、用户 query、以及可选的历史对话。**检索到的 chunk 总长**可能超过剩余预算，导致截断、关键信息丢失或超长拒绝，因此需要**控制与压缩**注入的上下文。

---

## 二、控制策略

- **预算分配**：事先为“检索上下文”预留 token 预算（如 4K），再根据 chunk 平均长度与模型限制决定**最多注入几个 chunk**（如 top-5 或 top-10）；检索阶段就只取 top-K，K 由预算反推。  
- **按相关性截断**：对检索+重排后的 chunk 按分数从高到低依次加入，**直到总 token 数接近预算**即停止，保证优先用最相关的片段。  
- **长度感知排序**：在相关性相近时优先选**更短**的 chunk，使同预算下能放入更多条，提高信息密度。

---

## 三、压缩策略

- **摘要**：对长 chunk 或用 LLM 做**摘要**再注入，只保留与 query 最相关的要点；适合“只要大意”的问答，有信息损失。  
- **选择性保留**：用模型或规则对 chunk 内**句子/句群**打分，只保留高分句；或按关键词、实体与 query 重叠度筛选。  
- **层次化**：先注入**简短摘要或标题**，若模型需要再按需请求“展开某段”；或第一轮用摘要、第二轮对点击的 chunk 拉全文。  
- **模型侧长上下文**：选用**长 context 模型**（如 128K），在成本与延迟可接受时放宽预算；仍建议对单 chunk 做合理长度与数量控制，避免噪声过多。

---

## 四、小结与面试要点

**小结**：RAG 受 context 限制，需预算分配、按相关性截断与长度感知排序；压缩可用摘要、选择性保留、层次化注入或长上下文模型；在“信息量”与“不超长、少噪声”之间权衡。

**面试要点**：能说清“预算分配 + 按相关性截断”；能举 1～2 种压缩方式（摘要、选句）；长 context 是可选手段而非唯一解。

---

## 记忆要点

1. 控制：预算分配、按相关性截断、长度感知排序。  
2. 压缩：摘要、选择性保留句子、层次化注入。  
3. 权衡：信息量 vs 长度与噪声；长 context 模型可放宽预算。
