# 第 129 题：RAG 可观测与调试

## 题目

RAG 系统如何做可观测性与调试？如何定位检索与生成问题？

---

## 一、可观测内容

- **请求级**：每个请求的 **request_id/trace_id**，贯穿检索、重排、生成全链路。  
- **检索阶段**：query（原始与改写后）、检索参数（top_k、filter）、**召回 doc/chunk id 与分数**、重排前后顺序、最终注入的 chunk 数量与总 token 数。  
- **生成阶段**：模型、max_tokens、实际 token 数、延迟、是否流式；若支持则记录是否触发“无相关文档”降级。  
- **结果**：最终回答、引用列表（chunk id 与来源）、用户/会话 id、时间戳。

---

## 二、日志与指标

- **结构化日志**：上述内容以 JSON 等形式写入日志，便于按 request_id、user_id、时间检索；敏感内容脱敏。  
- **指标**：检索延迟（P50/P99）、召回数、重排前后分数分布、生成延迟与 token 消耗、错误率；按数据源或租户聚合便于做 SLA 与成本分析。  
- **追踪**：用 OpenTelemetry、LangSmith 等做 **trace**，将“检索→重排→生成”拆成 span，便于看耗时分布与瓶颈。

---

## 三、调试与 Bad Case 分析

- **单请求复现**：根据 request_id 拉取**完整 trace**（query、召回列表与分数、注入的 context、模型输入与输出），在本地或测试环境**回放**，定位是检索不准、重排问题还是生成幻觉。  
- **检索分析**：看**召回 chunk 与 query 的相关性**（可人工或用 NLI 模型打分）；无相关召回→查分块与索引、query 改写；相关但排序差→查重排与融合。  
- **生成分析**：对比**注入的 context** 与**生成内容**，查是否忠实、是否引用错误或凭空编造；对应改进 prompt、后验校验或引用约束。  
- **A/B 与实验**：对“分块策略、检索路数、重排模型、prompt”等做 A/B，用**同一评估集**比较指标（召回率、忠实度、引用准确率、人工分），指导迭代。

---

## 四、小结与面试要点

**小结**：RAG 可观测需请求级 trace、检索与生成各阶段的关键数据与指标；调试依赖完整 trace 回放、检索与生成分步分析、以及 A/B 实验；便于快速定位检索差还是生成差。

**面试要点**：能说清“trace 贯穿检索与生成”；能举例如何根据 trace 区分检索问题与生成问题；指标与 Bad Case 分析要提到。

---

## 记忆要点

1. 可观测：request_id、检索/重排/生成各阶段数据、延迟与 token 指标、trace。  
2. 调试：完整 trace 回放、检索相关性分析、生成忠实度与引用分析。  
3. 迭代：A/B 实验与统一评估集。
