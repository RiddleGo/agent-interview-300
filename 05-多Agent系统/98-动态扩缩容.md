# 第 98 题：多 Agent 的动态扩缩容

## 题目

如何实现多Agent系统的动态扩缩容？

---

## 一、目标

根据**负载**（请求量、队列长度、延迟）自动**增加或减少** Agent 实例数，在保证 SLA 的前提下控制成本：高负载扩容、低负载缩容。

---

## 二、度量与触发

- **度量**：QPS、平均/ P99 延迟、任务队列长度、CPU/GPU 利用率、错误率等；可设阈值或基于预测。  
- **触发**：超过上限阈值扩容（如队列 > N 或延迟 > T）；低于下限阈值缩容（如利用率 < 某值）；可加冷却时间避免抖动。  
- **策略**：步进扩缩（每次 ±1 或 ±N 实例）；或按比例（如当前实例数的 20%）；最大/最小实例数限制。

---

## 三、实现要点

- **无状态**：Agent 实例**无状态**或状态外置（会话、上下文在 Redis/DB），才能随时增删实例。  
- **任务来源**：从**公共队列**拉取任务（pull），新实例加入即参与消费；缩容前需排空或迁移正在执行的任务。  
- **编排层**：在 K8s 上可用 **HPA**（Horizontal Pod Autoscaler）按 CPU/自定义指标扩缩；自建则用脚本或调度器根据指标调实例数。  
- **优雅缩容**：缩容时不再接新任务、等当前任务完成或超时后再下线，避免中断执行中任务。

---

## 四、多 Agent 特有注意

- **角色与能力**：若不同角色 Agent 分开部署，需**按角色分别**扩缩（如“写手”与“审核”独立伸缩）。  
- **协调者**：协调者通常少量固定或主备，一般不随负载线性扩缩；扩展的是 Worker 类 Agent。  
- **成本**：LLM 调用成本高时，扩缩可结合**预算与配额**（如总 QPS 上限），避免盲目扩导致成本飙高。

---

## 五、小结与面试要点

**小结**：动态扩缩容 = 基于负载度量与阈值触发、无状态+队列、编排层执行、优雅缩容；多 Agent 需按角色扩缩、协调者通常不随负载线性扩。

**面试要点**：能说清度量、触发、无状态与队列；K8s HPA 或自定义调度；缩容要优雅、多角色按角色扩缩。

---

## 记忆要点

1. 度量：QPS、延迟、队列长度、利用率；触发：阈值+冷却。  
2. 实现：无状态、队列 pull、编排层扩缩、优雅缩容。  
3. 多 Agent：按角色扩缩、协调者一般不线性扩。
