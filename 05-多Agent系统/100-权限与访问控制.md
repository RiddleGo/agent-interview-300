# 第 100 题：多 Agent 的权限与访问控制（RBAC/ABAC）

## 题目

多Agent系统中如何做权限与访问控制？RBAC 与 ABAC 的区别与适用场景？

---

## 一、要控制什么

- **资源**：哪些 Agent 或用户可访问哪些**数据、工具、API、其他 Agent**。  
- **操作**：读/写/执行/委托/管理；不同操作可设不同权限。  
- **边界**：Agent 间、Agent 与用户、Agent 与外部服务之间的**最小权限**，避免越权与滥用。

---

## 二、RBAC（Role-Based Access Control）

**思路**：权限不直接绑用户/Agent，而是绑**角色**；用户/Agent 赋予一个或多个角色，通过角色获得权限。

**要素**：角色（如“研究员”“审核”“管理员”）、权限（如“读文档”“调工具 A”“派发任务”）、用户/Agent 与角色的分配关系。

**优点**：易管理、易审计（谁是什么角色）；适合**固定职责**的 Agent（如按角色分工的多 Agent 系统）。  
**缺点**：细粒度依赖角色数量；动态属性（如“仅自己的任务”）需扩展。

**适用**：多 Agent 中按角色（PM、开发、测试）划分权限；企业内部按岗位授权。

---

## 三、ABAC（Attribute-Based Access Control）

**思路**：按**属性**决定是否允许访问：主体属性（谁）、资源属性（什么）、环境属性（时间、IP、风险等级）等；策略用规则表达（如“主体.role=审核 且 资源.type=文档 且 环境.time=工作时间”）。

**优点**：细粒度、可表达复杂条件（如“仅任务负责人可读该任务结果”）。  
**缺点**：策略复杂、性能与审计难度高。

**适用**：需要**动态、细粒度**控制的场景（如“仅委托方可看该委托结果”；多租户按租户+资源属性控制）。

---

## 四、多 Agent 实践

- **默认**：每个 Agent 仅有其**角色所需最小权限**（最小权限原则）；委托时仅传递任务所需信息，不扩散无关权限。  
- **委托链**：A 委托 B 时，B 的权限限于“完成该任务所需”，不自动获得 A 的全部权限；可带**临时凭证或 scope**。  
- **审计**：关键操作（访问敏感数据、调用外部 API、委托）记日志，便于溯源与合规。

RBAC 适合角色固定的多 Agent；ABAC 适合需要“按任务、按资源、按环境”细粒度控制的场景；二者可结合（角色+属性策略）。

---

## 五、小结与面试要点

**小结**：权限控制资源与操作；RBAC=按角色授权、易管理；ABAC=按属性与规则、细粒度；多 Agent 宜最小权限、委托限 scope、审计关键操作。

**面试要点**：能区分 RBAC 与 ABAC；多 Agent 中角色对应 RBAC、细粒度与委托对应 ABAC；最小权限与审计必提。

---

## 记忆要点

1. RBAC：角色→权限，易管理，适合固定角色。  
2. ABAC：属性+规则，细粒度，适合动态条件。  
3. 多 Agent：最小权限、委托限 scope、审计。
