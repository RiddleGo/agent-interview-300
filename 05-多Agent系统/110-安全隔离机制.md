# 第 110 题：多 Agent 的安全隔离机制

## 题目

多Agent系统如何设计安全隔离机制？

---

## 一、隔离维度

- **租户/用户隔离**：不同租户或用户的数据、会话、资源**互不可见、互不影响**；访问控制与数据按 tenant_id/user_id 隔离。  
- **Agent 间隔离**：单个 Agent **不能越权**访问其他 Agent 的私有数据或能力；仅通过约定接口或消息传递协作；权限最小化（见“权限与访问控制”）。  
- **执行环境隔离**：若 Agent 执行**代码或工具**，应在**沙箱**中执行（如容器、VM、受限进程），限制文件、网络、系统调用，防止恶意或错误代码影响主机与其他 Agent。  
- **网络与依赖隔离**：Agent 或服务仅能访问**白名单**内的下游；内部服务与对外 API 分离；敏感依赖（如密钥、内部 API）不暴露给不可信 Agent。

---

## 二、实现手段

- **身份与授权**：每个请求/任务带**身份**（用户、租户、Agent 标识）；资源访问前做**鉴权**（RBAC/ABAC），未授权即拒绝。  
- **数据隔离**：存储层按 tenant_id 分区或加密；查询与导出强制带租户过滤；敏感字段加密或脱敏。  
- **沙箱**：代码执行、工具调用放在**隔离环境**（容器、gVisor、专用进程）；限制 CPU/内存/网络；超时与 kill。  
- **网络策略**：K8s NetworkPolicy、服务网格或防火墙限制“谁可访问谁”；Agent 仅能调允许的端点。  
- **审计**：关键操作（跨租户访问、敏感数据访问、特权操作）**审计日志**，便于溯源与合规。

---

## 三、多 Agent 特有

- **委托边界**：A 委托 B 时，B 仅能拿到**该任务所需**的信息与权限，不能拿到 A 的全部上下文或密钥；委托带 scope 与过期。  
- **恶意或异常 Agent**：若某 Agent 被攻破或异常，通过**隔离**限制其能影响的范围（仅其可见的数据、仅其可调的下游）；结合熔断与监控快速发现并摘除。

---

## 四、小结与面试要点

**小结**：安全隔离包括租户/用户、Agent 间、执行环境、网络与依赖；手段有身份与授权、数据隔离、沙箱、网络策略与审计；多 Agent 强调委托边界与单点影响范围限制。

**面试要点**：能说出多维度隔离（租户、Agent、执行、网络）；沙箱与最小权限必提；多 Agent 强调委托 scope 与影响范围。
