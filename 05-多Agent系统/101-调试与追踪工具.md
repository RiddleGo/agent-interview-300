# 第 101 题：多 Agent 的调试与追踪工具

## 题目

多Agent系统有哪些调试与追踪工具？如何定位问题？

---

## 一、追踪（Tracing）

- **全链路 Trace**：一次请求从入口到各 Agent、工具调用、LLM 调用的**完整调用链**，带唯一 trace_id；可记录 span（子调用）、耗时、状态。  
- **实现**：OpenTelemetry、LangSmith/LangFuse、自定义 SDK 在 Agent 与工具层埋点；将 trace 发往后端（如 Jaeger、Zipkin、厂商控制台）做聚合与查询。  
- **价值**：定位“慢在哪一环”“哪步失败”“调用顺序是否符合预期”。

---

## 二、日志与结构化日志

- **统一格式**：JSON 结构化日志，包含 trace_id、agent_id、role、timestamp、level、message、必要 payload（如任务 id、工具名、错误码）。  
- **聚合**：日志集中到 ELK、Loki、云日志等，按 trace_id 或 agent_id 检索；可配合告警（错误率、超时）。  
- **敏感信息**：脱敏或截断，避免把完整 prompt/用户数据写入日志。

---

## 三、调试能力

- **单步与回放**：在开发/测试环境支持**单步执行**（每步人工确认）或**按历史 trace 回放**，复现问题。  
- **中间状态导出**：可导出某步的输入/输出、上下文、工具结果，便于离线分析或注入到新 run。  
- **Mock 与注入**：对 LLM、工具或下游 Agent 做 Mock，控制返回值，验证分支与异常路径。  
- **可视化**：用 LangSmith、LangFuse、或自建 UI 展示 DAG、消息流、各步耗时与 token 消耗。

---

## 四、多 Agent 特有

- **角色与消息流**：日志中区分“谁发给谁、什么类型的消息”（任务、结果、控制）；便于理解协作顺序与冲突。  
- **共识与冲突**：对投票、仲裁、冲突解决步骤打点，便于分析“为何最终选 A 而非 B”。  
- **成本与配额**：按 trace 或 agent 统计 token/调用次数，便于做成本归因与限流排查。

---

## 五、小结与面试要点

**小结**：调试与追踪依赖全链路 trace、结构化日志、单步/回放/Mock 与可视化；多 Agent 需区分角色与消息流、共识步骤与成本。

**面试要点**：能说出 trace、日志聚合、单步/回放、Mock；多 Agent 强调消息流与共识的可观测性。

---

## 记忆要点

1. 追踪：全链路 trace_id、span、耗时与状态。  
2. 日志：结构化、聚合、按 trace/agent 检索。  
3. 调试：单步、回放、Mock、可视化；多 Agent 看消息流与共识。
