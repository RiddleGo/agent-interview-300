# 第 107 题：多 Agent 的日志聚合与链路追踪

## 题目

多Agent系统如何做日志聚合与链路追踪？

---

## 一、链路追踪（Distributed Tracing）

**目标**：一次**请求/任务**穿过多个 Agent、服务、工具调用时，形成一条**完整调用链**，便于看耗时、依赖与故障点。

**做法**：
- **Trace / Span**：为每次请求分配 **trace_id**；每个子调用（如某 Agent、某工具、某 LLM 调用）为一个 **span**，带 parent_span_id、名称、开始/结束时间、状态、可选属性（如 agent_id、model、token 数）。  
- **传递**：trace_id（及当前 span_id）在跨进程调用时通过 header 或消息体传递，保证整链同 id。  
- **采集与存储**：SDK 或中间件将 span 发往后端（如 Jaeger、Zipkin、Tempo、或云厂商）；按 trace_id 聚合、按时间与服务查询。

多 Agent 中每个 Agent 的入口/出口、工具调用、LLM 调用都打 span，即可得到“谁调谁、多久”的完整视图。

---

## 二、日志聚合（Log Aggregation）

**目标**：把分散在各实例/容器上的**日志**集中存储与检索，便于按请求、按 Agent、按时间排查。

**做法**：
- **统一格式**：结构化日志（如 JSON），必含 trace_id、agent_id、timestamp、level、message；可选 request_id、user_id、task_id。  
- **采集**：每实例通过 agent/侧车将日志推到中心（如 Fluentd、Filebeat、云日志）；或写本地文件由日志平台采集。  
- **存储与检索**：写入 Elasticsearch、Loki、S3+检索等；按 trace_id 查即得“该请求涉及的所有日志”；按 agent_id、时间、错误级别筛选。

---

## 三、关联使用

- 日志中**带 trace_id**，可与 trace 系统关联：先查 trace 定位慢/错的 span，再用 trace_id 查日志看该链路的详细日志。  
- 多 Agent 场景下，可为**消息/任务**打 trace_id，整条协作链共用一 trace，便于看“从用户请求到多 Agent 协作完成”的全过程。

---

## 四、小结与面试要点

**小结**：链路追踪 = trace_id + span、跨进程传递、采集到后端聚合；日志聚合 = 结构化日志 + trace_id、集中采集与检索；二者通过 trace_id 关联，多 Agent 整链打同一 trace。

**面试要点**：能说清 trace/span、trace_id 传递、日志带 trace_id 与集中检索；多 Agent 强调整链一 trace。

---

## 记忆要点

1. 追踪：trace_id + span、跨进程传递、后端聚合。  
2. 日志：结构化、带 trace_id、集中采集与检索。  
3. 关联：用 trace_id 从 trace 跳到日志；多 Agent 整链一 trace。
