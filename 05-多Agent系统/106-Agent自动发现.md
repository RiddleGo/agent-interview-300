# 第 106 题：Agent 自动发现（Service Discovery）

## 题目

多Agent系统中如何实现Agent的自动发现（Service Discovery）？

---

## 一、要解决什么

分布式下，Agent 或服务**实例**会动态增删、换地址；调用方需要知道“**谁在、在哪、是否可用**”，而不写死 IP/端口。Service Discovery = **注册**（实例上报自身） + **发现**（调用方查询可用实例）。

---

## 二、常见实现

- **注册中心**：集中式存储“服务名 → 实例列表（地址、元数据、健康状态）”；实例**注册**（上线时）、**心跳**（定期保活）、**下线**（优雅关闭时注销）。例如 Consul、Etcd、Nacos、Zookeeper；K8s 用 Service + Endpoints 也可视为一种发现。  
- **DNS**：通过 DNS 查询服务名得到地址；可配合健康检查做摘除，适合简单场景。  
- **客户端/服务端发现**：**服务端发现** = 调用方请求负载均衡器/网关，由后者查注册中心并转发；**客户端发现** = 调用方直接查注册中心并自行选实例、调用。多 Agent 里“协调者选 Worker”常用客户端发现（协调者查注册中心选目标 Agent）。

---

## 三、与多 Agent 的对应

- **Agent 即服务**：每个 Agent 类型或角色对应一“服务名”；多实例时列出所有实例，再按负载、能力、策略选一个。  
- **元数据**：注册时带**能力标签**（如支持的工具、模型、租户），发现时按任务需求过滤，实现“能力匹配”的发现。  
- **健康与熔断**：发现结果结合健康检查；不健康实例不返回或降权；结合熔断避免反复选到故障实例。

---

## 四、小结与面试要点

**小结**：自动发现 = 注册（实例上报）+ 发现（查询可用实例）；常用注册中心 + 心跳 + 下线；多 Agent 可把 Agent 当服务、带能力元数据、结合健康与熔断。

**面试要点**：能说清注册与发现、客户端 vs 服务端发现；多 Agent 中 Agent 即服务、能力标签与健康结合。

---

## 记忆要点

1. 发现 = 注册 + 查询；注册中心 + 心跳 + 下线。  
2. 客户端发现：调用方查注册中心并选实例；多 Agent 常用。  
3. Agent 即服务、带能力元数据、结合健康与熔断。
