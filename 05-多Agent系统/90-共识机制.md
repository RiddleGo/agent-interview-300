# 第 90 题：多 Agent 中的共识机制（Consensus Mechanism）设计

## 题目

解释多Agent中的共识机制（Consensus Mechanism）设计。

---

## 一、共识要解决什么

多 Agent 在**无中心或弱中心**场景下，需要对某**结论、顺序或状态**达成一致（如“采用哪条计划”“谁先执行”“当前全局状态是什么”），避免各说各话与状态分裂。共识机制即**在分布式条件下达成一致**的规则与流程。

---

## 二、常见设计

- **多数投票**：对离散决策（如选 A/B/C）各 Agent 投票，取多数；实现简单，适合低延迟要求。  
- **加权投票**：按 Agent 权重、角色或置信度加权，再汇总；适合“专家权重大”等场景。  
- **Quorum**：规定“超过 N 个 Agent 同意才通过”，兼顾可用性与一致性。  
- **两阶段/多阶段**：先收集提案与投票，再提交（commit）或中止（abort），减少不一致提交。  
- **Leader 提议+确认**：由 Leader 提出方案，其他 Agent 确认；Leader 轮换可防单点。  
- **区块链式**：在需要不可篡改与顺序共识时，可引入链式结构或 PoS/PoW 类机制（多用于跨组织、高信任要求场景）。

---

## 三、与冲突解决的关系

共识常作为**冲突解决**的一种实现：通过投票或 Quorum 达成“采用哪个结果”；也可先局部共识（如小组内一致）再上层汇总。设计时需权衡**延迟、可用性、一致性**（如 CAP）。

---

## 四、小结与面试要点

**小结**：共识 = 多 Agent 对结论/顺序/状态达成一致；方式有投票、加权、Quorum、两阶段、Leader+确认等；需权衡延迟、可用性与一致性。

**面试要点**：共识目的（分布式一致）；举例（投票、Quorum、两阶段）；与冲突解决的关系（共识可作为冲突解决的实现）。

---

## 记忆要点

1. 共识 = 多 Agent 对结论/顺序/状态达成一致。  
2. 方式：多数/加权投票、Quorum、两阶段、Leader+确认。  
3. 权衡延迟、可用性、一致性。
