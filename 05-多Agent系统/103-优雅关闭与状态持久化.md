# 第 103 题：多 Agent 的优雅关闭与状态持久化

## 题目

如何实现多Agent的优雅关闭（Graceful Shutdown）与状态持久化？

---

## 一、优雅关闭（Graceful Shutdown）

**目标**：进程/实例下线时，**不中断**正在执行的任务、不丢消息、不让新请求再进来，再安全退出。

**要点**：
- **停止接收**：收到关闭信号（如 SIGTERM）后，不再接新任务或新请求；若从队列拉任务，则不再 poll。
- **排空进行中**：等待当前正在执行的任务**完成**（或超时）；可设最大等待时间，超时后强制退出并记录未完成任务。
- **通知与协调**：在分布式下通知负载均衡/注册中心“本实例正在关闭”，不再被选为新请求目标；必要时通知上游“我将下线”。
- **资源释放**：关闭连接池、文件句柄、订阅等，再退出进程。

多 Agent 中每个 Agent 实例、协调者、Worker 都应按上述方式优雅关闭，避免任务执行到一半被 kill。

---

## 二、状态持久化

**目标**：把**会话、上下文、任务状态、未完成工作**等存到外部存储，以便重启、扩缩容或故障恢复后**续跑**。

**要持久化的内容**：
- **任务状态**：任务 id、状态（pending/running/done/failed）、输入、已产生的输出、归属 Agent。
- **会话/上下文**：多轮对话历史、当前轮次、共享黑板中的关键状态。
- **检查点**：长流程的中间检查点（如“已完成步骤 1～3”），便于从某步恢复。

**实现**：用 DB（如 PostgreSQL）、Redis、对象存储等；按 task_id/session_id 读写；写入时机可为“每步结束”或“每 N 步/每 N 秒”做 checkpoint；恢复时根据 task_id 加载状态并从中断处继续。

---

## 三、结合使用

- 优雅关闭前，将**当前任务状态与上下文**持久化，标记“实例关闭、任务可被其他实例接管或稍后恢复”。  
- 新实例或重启后，可从持久化存储**拉取未完成任务**并恢复执行（需任务幂等或带版本，避免重复执行副作用）。

---

## 四、小结与面试要点

**小结**：优雅关闭 = 停接新请求、排空进行中、通知与释放资源；状态持久化 = 任务状态、会话/上下文、检查点存外部存储，支持恢复与续跑；二者结合可做到无中断下线与故障恢复。

**面试要点**：能说清“停接新、排空、通知、释放”；持久化内容与恢复流程；多 Agent 每实例都需优雅关闭。

---

## 记忆要点

1. 优雅关闭：停接新、排空进行中、通知、释放资源。  
2. 状态持久化：任务状态、会话/上下文、检查点；支持恢复与续跑。  
3. 结合：关闭前持久化，恢复时从未完成任务继续。
