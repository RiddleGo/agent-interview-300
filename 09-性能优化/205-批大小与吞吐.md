# 第 205 题：批大小与吞吐

## 题目

批大小（Batch Size）如何影响 LLM 推理吞吐与延迟？如何选择？

---

## 一、批大小与吞吐

- **大 batch**：**单步**算的 **token 数**多，**GPU 利用率**高，**吞吐**（tokens/s 或 requests/s）常**随 batch 增大**而升；但**显存**与**单请求延迟**（排队+算）会**上升**。  
- **小 batch**：**延迟**好（排队短、TTFT 小），但**利用率**可能不足，**吞吐**低。  
- **拐点**：**过大** batch 时**显存**打满、**无法**再增，或** Kernel** 效率下降；**吞吐**曲线会有**拐点**；**最优** batch 需**实测**（压测+监控）。

---

## 二、与延迟的权衡

- **静态批**：**固定** batch 时，**大 batch**→**排队**等凑批→**TTFT 差**；**小 batch**→**TTFT 好**但**吞吐**低。  
- **连续批**：**动态** batch，**不凑满**即可算；**长短不一**时**先结束先出**；**兼顾**吞吐与 **TTFT**，**无需**手动定“一个” batch 值，由**调度器**与**到达率**决定。  
- **Prefill vs Decode**：**Prefill** 算力/带宽重，**大 batch** 利于**吞吐**；**Decode** 每步 token 少，**过大** batch **收益**递减；可**分离** prefill 与 decode 的**批策略**（如 prefill 大 batch、decode 连续批）。

---

## 三、选择与实践

- **目标**：**高吞吐**（离线、批量）→**大 batch**；**低延迟**（在线）→**小 batch** 或**连续批**。  
- **约束**：**显存**上限→**batch×seq** 有上界；**SLA**（P99 延迟）→**batch** 不宜过大导致排队爆表。  
- **调参**：**压测**不同 **batch**（或**最大 batch**）下的**吞吐**与 **P99**；**选**满足 **SLA** 的**最大** batch 或**启用连续批**。  
- **多模型**：**不同模型** **显存**与**算力**不同，**最优 batch** 不同；**分别**调或**分别**配置。

---

## 四、小结与面试要点

**小结**：批大小影响吞吐与延迟；大 batch 吞吐高、延迟差，小 batch 反之；连续批可兼顾；需结合显存与 SLA 实测选 batch；Prefill 与 Decode 可分离策略。

**面试要点**：能说清 batch 与吞吐、延迟的关系；连续批要提到；显存与 SLA 约束要强调。

---

## 记忆要点

1. 大 batch：吞吐高、延迟差、显存增；小 batch 反之。  
2. 连续批：动态、兼顾吞吐与 TTFT。  
3. 选择：目标（吞吐 vs 延迟）、显存、SLA；实测调参。
