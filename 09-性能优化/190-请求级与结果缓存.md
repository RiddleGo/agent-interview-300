# 第 190 题：请求级与结果缓存

## 题目

请求级缓存与结果缓存的区别与实现要点？

---

## 一、请求级缓存

**含义**：以**单次请求**（通常为 **query/prompt** 或归一化后的 hash）为粒度，**命中则直接返回**当时缓存的**完整响应**，**不再**调 LLM、检索或工具。

**特点**：**实现简单**（key=query hash，value=response）；**延迟与成本**最优（无推理）；**命中率**依赖**请求重复度**（FAQ、固定话术高，开放对话低）。  
**注意**：**多轮**时 key 需含**会话上下文**（如最近 N 轮 hash）或**会话 id + 当前 query**；**模型/配置/数据**变更时需**失效**相关 key。

---

## 二、结果缓存（中间结果）

**含义**：缓存**链路中间结果**，而非整条响应。例如：  
- **Embedding**：相同文本的 embedding 只算一次，**key=text hash**。  
- **检索结果**：相同 query 的 **top-k chunk**；**key=query hash**（或 embedding）；可带 **filter** 参与 key。  
- **工具/API**：相同参数的**调用结果**；**key=工具名+参数 hash**；**TTL** 按数据新鲜度（如天气 1 小时）。  
- **子模型输出**：若**多步推理**（如先分类再生成），可缓存**分类结果**等。

**特点**：**单步**命中即可**减少该步**计算或调用；**组合请求**仍可能**部分命中**（如 embedding 命中、LLM 未命中）；**粒度细**，**失效**可更精准（如只清某数据源相关的检索缓存）。

---

## 三、实现要点

- **Key**：**请求级**用 **hash(prompt)** 或 **hash(tenant_id + model + prompt)**；**结果级**用 **步骤+输入 hash**（如 `emb:v1:hash(text)`）。  
- **存储**：**响应**可能较大，可**压缩**或**存对象、key 存引用**；**中间结果**（向量、JSON）注意**序列化开销**。  
- **失效**：**请求级**随 **model/version/prompt 模板** 失效；**结果级**随**数据源更新**（如文档更新清检索缓存）、**TTL** 或**版本**失效。  
- **监控**：**命中率**（请求级、各结果级）；**节省的 QPS/Token**；**误命中**（语义变但 key 同）需抽样检查。

---

## 四、小结与面试要点

**小结**：请求级缓存=整请求整响应，命中则不再调 LLM；结果缓存=中间步骤（Embedding、检索、工具）缓存，部分命中也可减延迟与成本；两者可并存；Key、失效与监控需设计好。

**面试要点**：能区分请求级与结果级；多轮与 key 设计要提到；失效与命中率监控要举一二。

---

## 记忆要点

1. 请求级：整请求→整响应；命中即不调 LLM；依赖重复度。  
2. 结果级：Embedding/检索/工具等中间结果；部分命中也有效。  
3. Key、失效、多轮 key、命中率监控。
