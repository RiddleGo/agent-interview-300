# 第 189 题：缓存策略与实现

## 题目

LLM/Agent 应用中有哪些缓存策略？如何实现？

---

## 一、缓存对象

- **完整响应**：**相同请求**（如同 query、同 prompt）直接返回**历史响应**；实现简单，**命中依赖** query 重复度。  
- **中间结果**：**Embedding** 结果、**检索 top-k**、**Reranker 结果**；相同或**相似 query** 可复用，减少**重复计算与下游调用**。  
- **KV Cache**：**相同 prefix**（system、长 context）的 **KV Cache**；下次请求**跳过**该段 prefill，降低 **TTFT** 与算力。  
- **工具/API 结果**：**天气、汇率、通用查询**等**外部调用**结果；按 **key**（如 query 或参数）缓存，**TTL** 控制新鲜度。

---

## 二、Key 设计

- **精确**：**原始 query** 或 **hash(prompt)** 为 key；**命中**即完全一致；适合**重复度高**的 FAQ、固定流程。  
- **归一化**：**去空格、小写、trim** 后再 hash；**略提高命中**，注意**语义是否仍等价**。  
- **语义**：**Embedding** 后**向量相似**的 query 共享缓存（见“语义缓存”）；**key** 可为**聚类 id** 或**代表向量**；需**相似度阈值**与**更新策略**。  
- **分层**：**多级 key**（如 tenant_id + model_id + hash(prompt)）；**隔离**与**失效**按需（如按租户、按模型版本）。

---

## 三、实现要点

- **存储**：**内存**（单机、快）、**Redis**（分布式、共享）、**本地+Redis** 分层；**大 value**（长回复）可**压缩**或**存 OSS、key 存引用**。  
- **失效**：**TTL**（如 1 小时、1 天）；**版本/配置变更**时**主动失效**（如 model 升级清缓存）；**写穿透**（更新数据时删或更新缓存）。  
- **一致性**：**缓存与源**可能短暂不一致；**读时**可接受**最终一致**或**短 TTL**；**关键业务**可**先更 DB 再删缓存**。  
- **成本与容量**：**容量与淘汰**（LRU、LFU）；**成本**（Redis 内存、带宽）；**监控**命中率、大小与延迟。

---

## 四、小结与面试要点

**小结**：缓存对象包括完整响应、Embedding/检索、KV Cache、工具结果；Key 可精确、归一化或语义；实现需存储选型、失效策略、一致性与容量管理；命中率与成本需监控。

**面试要点**：能列举 2～3 类缓存对象；Key 设计（精确 vs 语义）要提到；TTL 与失效、存储分层要举一二。

---

## 记忆要点

1. 对象：完整响应、Embedding/检索、KV Cache、工具结果。  
2. Key：精确、归一化、语义（相似 query）；分层 key 做隔离。  
3. 实现：存储（内存/Redis）、TTL 与主动失效、一致性、容量与监控。
