# 第 202 题：检索与 RAG 延迟

## 题目

RAG 链路中检索阶段延迟如何分析与优化？

---

## 一、检索延迟组成

- **Query 处理**：**改写、扩展、HyDE** 等若用 **LLM**，会**增加**一次模型调用；**规则**或**小模型**可缩短。  
- **Embedding**：**Query** 的 **embedding** 计算；**批量**与**模型选型**（小/量化）影响耗时；**缓存**相同/相似 query 的 embedding。  
- **向量检索**：**ANN** 在向量库中的**搜索**；**索引类型**（HNSW、IVF）、**数据量**、**top_k**、**过滤**条件影响；**本地/近端**部署减少 RTT。  
- **重排（Reranker）**：对 **top-N** 做**精排**；**Cross-Encoder** 需** N 次**前向；**N 不宜过大**（如 20～50），或**异步/并行**。  
- **组装**：**Chunk** 取回、**拼接**、**截断**；通常占比小；**大 chunk** 可**只传引用**，按需拉正文。

---

## 二、优化手段

- **缓存**：**Query embedding** 与**检索结果**（精确或语义）缓存；**高重复**场景**命中**即跳过检索。  
- **并行**：**Embedding** 与**上游**（如 query 改写）**并行**；**多路检索**（稠密+稀疏）**并行**再融合；**检索**与**非依赖**的其它步骤**并行**。  
- **减少量**：**top_k** 不宜过大；**重排**只对**精简后**的候选做；**早停**（如已有高相关则不再扩召）。  
- **索引与部署**：**索引**选**低延迟**算法与参数；**向量库**与**应用**同可用区；**只读副本**分担查询。  
- **预计算**：**热点 query** 或**定时任务**预计算** embedding 与 top_k**，**请求时**直接读；适合**推荐、固定问答**。

---

## 三、与生成的关系

- **检索与生成**：**检索**可与**首 token 前**的** prompt 组装**重叠（检索完成即注入）；**流式**下**首 token** 仍受**检索完成时间**约束；**优化检索**即**优化** RAG 的 **TTFT**。  
- **超时**：**检索**设**超时**；超时则**用缓存**或**无检索**降级，避免**整请求**被拖死。

---

## 四、小结与面试要点

**小结**：检索延迟含 query 处理、embedding、向量检索、重排与组装；优化包括缓存、并行、减少 top_k 与重排量、索引与部署、预计算；检索影响 RAG 首 token，需设超时与降级。

**面试要点**：能拆解检索各阶段；能举 2～3 个优化（缓存、并行、减量）；与生成的关系要提到。

---

## 记忆要点

1. 组成：query 处理、embedding、向量检索、重排、组装。  
2. 优化：缓存、并行、减 top_k/重排量、索引与部署、预计算。  
3. 与生成：检索影响 TTFT；超时与降级。
