# 第 76 题：流式响应与打字机效果

## 题目

单 Agent 如何实现流式响应与“打字机”效果？有什么注意点？

---

## 一、为什么需要流式？

LLM 生成完整回复可能需要数秒甚至更久，若等全部生成再返回，用户会长时间无反馈，体验差。**流式响应**即模型每生成一段（如一个 token 或一小段）就立即推给前端，前端逐字或逐句展示，形成“打字机”效果，**首字延迟**明显降低，主观体验更顺畅。

---

## 二、实现要点

**后端**：调用 LLM 时使用**流式 API**（如 OpenAI SSE、服务端 streaming），按 chunk 写入响应体；若中间有工具调用，可在流式前先完成规划与工具调用，再对“最终回答”部分流式输出；或多段流式（工具前一段说明、工具后一段结果）。**协议**：常用 SSE（Server-Sent Events）或 WebSocket，便于前端订阅与解析。**前端**：订阅流式端点，收到 chunk 即追加到 UI，可加简单动画（如光标闪烁）强化打字机感。**会话与多轮**：流式响应需与 session 关联，避免多 tab 或并发请求串流；结束时明确标记 stream end，便于前端关闭连接或切到下一轮。

---

## 三、注意点

**与规划/工具的关系**：若整轮需“先规划→再工具→再生成”，则流式主要针对最后生成阶段；也可在“等待工具”时先流式输出“正在查询…”等中间态。**错误与重试**：流式中途失败时，需能部分展示已生成内容并提示错误，或支持重试。**安全与脱敏**：流式内容同样需做输出过滤与脱敏。**指标**：首 token 时间（TTFT）与完整响应时间分开监控，便于优化体验与容量。

---

## 小结与面试要点

**小结**：流式响应通过 LLM 流式 API 按 chunk 推送到前端，实现打字机效果、降低首字延迟；实现需后端流式接口、SSE/WebSocket、前端逐字展示，并处理好与规划/工具、错误、安全及指标。

**面试要点**：目的=降低首字延迟、提升体验；实现=流式 API+SSE/WS+前端追加；注意规划/工具顺序、错误、脱敏与 TTFT 指标。

---

## 记忆要点

1. 流式目的：降低首字延迟，打字机效果，提升主观体验。  
2. 实现：后端流式 API、SSE/WebSocket、前端按 chunk 追加；与规划/工具阶段配合。  
3. 注意：会话关联、stream end 标记、错误与重试、脱敏、TTFT 指标。
