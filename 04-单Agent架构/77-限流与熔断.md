# 第 77 题：限流与熔断

## 题目

单 Agent 系统如何设计限流与熔断？分别解决什么问题？

---

## 一、限流（Rate Limiting）

**目的**：控制单位时间内的请求量或资源使用（如 QPS、并发数、单用户调用次数），防止系统过载、下游被打爆或成本失控。**常见策略**：**固定窗口**（如每分钟 N 次）、**滑动窗口**、**令牌桶/漏桶**，按 IP、用户、API Key 或接口维度限流。**实现**：网关或应用层计数，超限返回 429 或排队；可结合 Redis 等做分布式计数。**与 Agent 的关系**：对入口请求限流、对单用户/会话的模型或工具调用限流，避免单用户占满资源；对调用下游工具也需限流，避免把第三方或内部服务打挂。

---

## 二、熔断（Circuit Breaker）

**目的**：当下游服务**持续失败或超时**时，快速失败、不再持续请求，避免雪崩与资源耗尽，并给下游恢复时间。**状态**：通常为**关闭**（正常请求）→**打开**（失败达阈值后，直接拒绝一段时间）→**半开**（试探放行少量请求，成功则关闭，失败则再打开）。**参数**：失败次数/比例阈值、打开持续时间、半开试探量。**与 Agent 的关系**：对每个下游工具或 LLM 服务维护熔断器，打开时走降级（缓存、默认回复、报错提示）或换备用；避免单点故障拖死整条 Agent 链路。

---

## 三、配合使用

限流在**入口与调用侧**控制流量与成本；熔断在**下游异常**时保护系统与下游。二者可叠加：先限流控制总量，再对每个下游做熔断；并配合**监控与告警**（限流触发率、熔断打开次数）做容量与稳定性治理。

---

## 小结与面试要点

**小结**：限流控制 QPS/并发与成本，策略有固定/滑动窗口、令牌桶等，按用户或接口维度；熔断在下游持续失败时快速失败并降级，状态为关闭→打开→半开；二者配合保护系统与下游，需监控与告警。

**面试要点**：限流=控流量与成本，策略与维度；熔断=下游故障时快速失败与降级，三态；配合使用与监控。

---

## 记忆要点

1. 限流：控 QPS/并发/单用户用量；策略=窗口/令牌桶；按用户或接口维度。  
2. 熔断：下游持续失败时打开，直接拒绝或降级；关闭→打开→半开。  
3. 配合：限流控入口与调用，熔断保下游；需监控限流触发率与熔断打开情况。
