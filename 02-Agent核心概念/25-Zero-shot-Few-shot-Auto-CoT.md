# 第 25 题：Zero-shot、Few-shot、Auto-CoT 在 Agent 任务中的表现

## 题目

对比 Zero-shot、Few-shot、Auto-CoT 在 Agent 任务中的表现。

---

## 一、三者的含义（在 Agent 语境下）

- **Zero-shot**：不提供任何任务示例或推理示范，只给任务描述和工具描述，让模型直接做推理与工具选择。  
- **Few-shot**：在 prompt 中提供少量“任务→推理→工具调用→观察→答案”的完整示例，让模型模仿格式与策略。  
- **Auto-CoT**：自动生成或抽取 CoT/ReAct 式示例（如用“Let's think step by step”让模型自己生成推理链，再从中选作 few-shot 示例），再用于后续查询。

在 Agent 场景下，“表现”通常指：任务完成率、工具调用正确率、步数/效率、以及是否稳定产生合理推理与行动序列。

---

## 二、表现对比（定性）

| 方式 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **Zero-shot** | 无需标注、prompt 短、成本低 | 格式易漂移、工具选择不稳定、多步任务易漏步或乱序 | 简单任务、工具少、模型已较强 |
| **Few-shot** | 格式稳定、工具选用与顺序更可控、多步表现更好 | 占 context、需构造示例、示例质量敏感 | 复杂多步、工具多、需稳定格式 |
| **Auto-CoT** | 少人工标注、可自动得到推理链示例 | 自动示例质量参差、可能放大模型已有偏见、计算成本高 | 无现成示例时做“自举”增强 |

**一般结论**：  
- **简单 Agent 任务**（单步工具、明确意图）：Zero-shot 在大模型上常够用。  
- **复杂多步、多工具**：Few-shot 明显更稳，示例中最好覆盖“推理→选工具→处理观察→再推理”的完整链。  
- **Auto-CoT**：在无标注时可用来自动生成 few-shot 候选，但需筛选或校验，避免错误示例拉低表现。

---

## 三、面试中如何答

- **Zero-shot**：零示例，只靠任务与工具描述；实现简单但复杂任务上容易格式不稳、漏步。  
- **Few-shot**：给 1–3 个完整示例（含推理+工具+观察），显著提升格式与工具选择稳定性，尤其多步任务。  
- **Auto-CoT**：用模型自己生成推理链再当示例；可减少人工写示例，但要控制质量和多样性。  
- 实际系统常**混合**：简单路径 zero-shot，复杂路径 few-shot 或从 Auto-CoT 筛出的示例。

---

## 四、小结与面试要点

**小结**：Zero-shot 简单但复杂任务不稳；Few-shot 显著提升多步与工具选择稳定性；Auto-CoT 可自动产生示例但需质量控制；复杂 Agent 任务常用 Few-shot 或混合策略。

**面试要点**：

- 三者区别：Zero-shot 无示例；Few-shot 给完整示例；Auto-CoT 用模型生成示例再复用。  
- Agent 任务上：Few-shot 对多步、多工具更稳；Zero-shot 适合简单单步；Auto-CoT 可作为示例来源但需筛选。  
- 实践：复杂任务多用 Few-shot，示例要含推理+工具+观察的完整链。

---

## 记忆要点

1. Zero-shot：无示例，简单省事，复杂任务易不稳。  
2. Few-shot：给完整示例，格式与工具选择更稳，适合多步 Agent。  
3. Auto-CoT：自动生成推理链当示例，需质量控制；可作 Few-shot 的补充来源。
